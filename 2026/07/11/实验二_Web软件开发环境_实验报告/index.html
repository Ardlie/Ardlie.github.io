实验二 Web软件开发环境

Posted on 2026-07-11 Edited on 2026-07-12

1、实验目的

  1. 安装并运行一个Web博客软件（本实验选用 Hexo 静态博客框架）；
  2. 了解和掌握Web开发环境的基础安装与配置（Node.js、npm、Git）；
  3. 理解Web软件的B/S架构特点以及静态站点与动态站点的区别。

2、实验环境

  • 操作系统：Windows 10/11
  • 所需软件：
    • Node.js（LTS版本，含npm）
    • Git for Windows
    • Hexo CLI
    • VS Code

3、实验过程

步骤一：安装Node.js与npm

  1. 打开浏览器，访问 Node.js 官网：https://nodejs.org/
  2. 下载 LTS（长期支持）版本 的Windows安装包。
  3. 双击下载的安装包，启动安装向导。
  4. 在安装过程中，务必勾选”Automatically install the necessary tools”，并确保 “Add to PATH” 选项被选中。
  5. 其余步骤保持默认，点击”Next”直至安装完成。

步骤二：安装Git for Windows

  1. 打开浏览器，访问Git官网：https://git-scm.com/download/win
  2. 下载适合你系统的安装包。
  3. 双击安装包，启动安装向导。
  4. 在安装过程中，当出现 “Select Components” 页面时，建议勾选 “Git Bash Here” 和 “Git GUI Here” 。
  5. 在 “Choosing the default editor” 页面，可以选择VS Code（如果已安装）或其他编辑器。
  6. 在 “Adjusting your PATH environment” 页面，选择 “Git from the command line and also from 3rd-party software” 。
  7. 其余步骤保持默认，点击”Next”直至安装完成。

5、知识总结

一、核心概念

概念 说明
B/S 架构 Browser/Server（浏览器/服务器）架构，用户通过浏览器访问 Web 服务，无需安装客户端软件
静态站点 所有页面在部署前已预先生成好 HTML 文件，访问时直接返回，速度快、无需数据库
动态站点 页面在用户请求时由服务器实时生成（需查询数据库、渲染模板），交互性强但速度相对较慢
Hexo 基于 Node.js 的静态博客框架，支持 Markdown 写作，一键生成静态 HTML 页面
Markdown 轻量级标记语言，用简单语法（如 # 表示标题）书写纯文本，可转换为 HTML
Front-matter Markdown 文件开头的 YAML 格式元信息区域，用 --- 包裹，用于定义文章的标题、日期、标签等
CLI Command-Line Interface（命令行界面），通过输入命令来操作软件的工具

二、核心命令

命令 简写 功能 使用场景
node -v 无 查看 Node.js 版本号 验证 Node.js 是否安装成功
npm -v 无 查看 npm 版本号 验证 npm 是否安装成功
git --version 无 查看 Git 版本号 验证 Git 是否安装成功
npm install -g hexo-cli 无 全局安装 Hexo CLI 工具 第一次安装 Hexo 命令行工具
hexo init <文件夹名> 无 初始化博客项目 新建一个博客项目
cd <目录路径> 无 切换当前工作目录 进入项目文件夹
npm install 无 安装项目依赖包 在项目目录下安装所需的所有依赖
hexo server hexo s 启动本地预览服务 在本地查看博客效果
hexo generate hexo g 生成静态 HTML 页面 将 Markdown 文章编译成静态文件
hexo new "<文章标题>" hexo n "<文章标题>" 创建一篇新文章 新建博客文章
hexo clean 无 清除缓存和已生成的文件 重新生成前清理旧文件
hexo deploy hexo d 部署到远程服务器 将静态文件发布到 GitHub Pages 等
Ctrl + C 无 停止当前运行的进程 停止 hexo server 服务

三、配置文件（项目目录结构）

文件/文件夹 类型 作用
_config.yml 文件 🌐 站点主配置文件，定义博客标题、URL、主题、部署设置等
package.json 文件 📦 项目依赖清单，记录所有需要安装的 npm 包及版本
package-lock.json 文件 🔒 锁定依赖包的精确版本，确保团队环境一致
node_modules/ 文件夹 📁 存放所有已安装的依赖包（由 npm install 生成）
scaffolds/ 文件夹 📝 文章模板目录，新建文章时以此为模板生成
scaffolds/post.md 文件 默认文章模板（Front-matter 格式）
source/ 文件夹 📂 内容源文件目录，存放所有原始内容
source/_posts/ 文件夹 📄 文章存放目录，所有 .md 博客文章都放在此处
source/_drafts/ 文件夹 📄 草稿存放目录（未发布的草稿文章）
themes/ 文件夹 🎨 主题目录，存放博客的界面主题
themes/landscape/ 文件夹 默认主题（landscape），可替换为其他主题
.gitignore 文件 Git 版本控制忽略文件列表（如 node_modules/）

6、出现问题

序号 问题 严重程度 解决方案
1 hexo 命令无法识别 ⚠️ 中 重新安装 Hexo CLI 或使用 npx
2 hexo init 卡住 ⚠️ 中 使用 Gitee 镜像或 --no-clone 参数
3 npm install 安装缓慢 ⚠️ 低 切换国内镜像源（npmmirror）
4 allow-scripts 安全警告 ℹ️ 低 批准执行或忽略
5 localhost:4000 无法访问 🔴 高 确认服务启动/更换端口
6 新文章首页不显示 ⚠️ 中 执行 hexo generate 后刷新

7、心得体会

技术层面：实践是最好的学习方式

通过本次实验，我亲身体验了 Web 开发环境的完整搭建流程。从 Node.js 安装到 Hexo 博客上线，每一步都需要细心和耐心。印象最深的是 hexo init 因网络问题卡住时的不知所措，以及最终通过切换镜像源成功安装的成就感。

这让我认识到：一个完整的 Web 项目，不仅需要代码能力，更需要对工具链的熟悉和问题排查能力。遇到问题不再害怕，而是学会分析日志、搜索解决方案、逐步验证，这本身就是宝贵的学习过程。

课程思政层面：软件开发中的工匠精神

本次实验中，Hexo 框架的模块化设计给我留下了深刻印象：

  • source/ 存放内容，themes/ 控制外观，_config.yml 统一配置——职责清晰、分工明确
  • 依赖包通过 package.json 统一管理，版本可控、便于协作

这种设计理念反映了软件工程中的 “高内聚、低耦合” 原则，也让我联想到中国传统文化中 “各司其职、协作共赢” 的智慧。一个优秀的软件系统，需要每个模块在各自领域做到极致，再通过良好的接口协同工作。这与社会发展同理——每个人在自己的岗位上精益求精，整个社会才能高效运转。
